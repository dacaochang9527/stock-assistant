## 背景

0527D3 中用户实际买入：

| 代码 | 名称 | 买入价 | 数量 | 金额 |
|---|---:|---:|---:|---:|
| 600863 | 华能蒙电 | 6.170 | 600 | 3702.00 |
| 603303 | 得邦照明 | 25.440 | 200 | 5088.00 |

问题由此产生：原先把 `0528D4` 近似等同为未买入强势观察，但当前规则已收敛：D4 不再找未买入参与机会，只管理已有持仓。

## 当前确认共识

### 1. D3 统一使用“观察”，不再使用“候选”

用户已确认：候选和观察含义接近，统一为“观察”。后续用户可见概念不再出现“候选区”。

统一命名：

```text
0528D3观察区
```

工程字段统一使用：

```text
pool_type = watch
```

不再使用：

```text
candidate
候选区
```

如果历史文件名或旧文本里出现 `candidate`，只视为旧遗留/生成原因，不作为新概念继续使用。

### 2. D3 是否需要分区：需要

这里定稿：**D3 也需要分区**。

原因不是因为 D3 有多个策略池，而是因为买入动作会改变提醒目标：

- 未买入时：关注是否进入买点区、是否回收观察价、是否跌破失效；
- 已买入后：关注成本、浮盈浮亏、止损、防守线、尾盘是否处理。

因此，D3 内部也应分：

```text
D3观察区：未买入，寻找买点/回收/失效
D3持仓区：已买入，管理当天持仓风险
```

这不是“候选 vs 买入”，而是：

```text
观察 vs 持仓
```

### 3. 买入后只保留“持仓事实”，不预设 D4/D5 策略

买入后不再设未买入观察区；持仓的退出/加仓策略（原 D4 持有区/D5 强制退出口径）验证期已移除，不再预设。

```text
持仓：已买入，仅记录事实（成本、数量、可卖性、失效位）；动作规则待验证后另定
```

未买入的 D3 观察票次日不转为持仓。

## 明日监控池换日硬规则

明天 `0528` 的盘中提醒不得继续使用 `0527D3` 的观察或持仓数据。

固定规则：

```text
08:50 切池：生成/写入 0528D3 / 0528D4 的 active watchlist
09:05 守门：确认 active watchlist 里的 stage 全部是 0528D3 或 0528D4
09:25 盘中 watchdog：再次检查 stage 日期前缀，不匹配则暂停本轮提醒并输出错误
```

也就是说，即使 08:50 或 09:05 任务失败，盘中 watchdog 也不能静默继续使用：

```text
0527D3观察区
0527D3持仓区
```

## 落盘 CSV 如何区分 D3/D4 与分区

### 1. 源文件命名

源文件仍按日期+D几和时间戳区分：

```text
0528D3_<原因>_HHMMSS.csv
0528D4_<原因>_HHMMSS.csv
```

`<原因>` 只说明来源，不参与优先级，例如：

```text
scan
manual_review
position_rollover
after_close_review
```

优先级只看：

```text
0528D3 / 0528D4 + HHMMSS
```

### 2. active watchlist 字段区分

最终盘中实际读取的是：

```text
data/watchlists/tulong_active_watchlist.csv
```

关键字段：

```text
stage       # 0528D3 / 0528D4
pool_type   # watch / position
source_file # 来自哪个源CSV
```

因此三类池可以在同一个 active CSV 中区分：

| 用户概念 | stage | pool_type |
|---|---|---|
| 0528D3观察区 | 0528D3 | watch |
| 0528D3持仓区 | 0528D3 | position |
| 0528D4持有区 | 0528D4 | position |

### 3. 持仓字段

持仓区额外需要：

```text
entry_date
entry_stage
entry_price
quantity
sellable_quantity
cost_amount
position_status
```

## 持仓的事实层差异：可卖性（验证期口径）

唯一保留的事实差异是 **T+1 可卖性**，它由 `entry_date` 客观决定，与具体策略无关：

```text
买入当天：T+1 限制下不能卖（sellable_quantity=0）
次日及以后：已解禁可卖（sellable_quantity=quantity）
```

| 维度 | 买入当天持仓 | 次日及以后持仓 |
|---|---:|---:|
| pool_type | position | position |
| 能否卖（事实） | 否（T+1 锁定） | 是 |

> 验证期移除：原“D3持仓区/D4持有区提醒指标、动作语言（止盈/止损/减仓/加仓/留仓/隔夜风险等）”属于尚未验证的**决策策略**，已整体移除，不再预设。
> 待积累足够持仓生命周期数据、验证出稳定的退出/加仓规则后，再回填具体提醒口径。当前仅呈现成本、现价、相对盈亏、距失效位等**事实指标**，判断权交由人工与复盘。

## 建议概念模型

### 总结构

```text
D3
├─ D3观察区：未买入，寻找买点/回收/失效（选股链路）
└─ 持仓（已买入）：进入持仓事实层，按 T+1 可卖性区分；具体退出/加仓策略验证期不预设
```

说明：买入后不再用 D4/D5 等日历阶段绑定固定策略。持仓只记录事实（成本、数量、可卖性、失效位），动作判断验证期交由人工与复盘。

### 用户可见命名

```text
0528D3观察区   # 未买入
0528D3持仓区   # 已买入，当天 T+1 不可卖（事实）
持仓（次日及以后）  # 已买入，已解禁可卖（事实）；不再固定命名为 D4/D5 持有区
```

### 工程字段建议

`stage` 记录入场时的选股阶段（如 `0527D3`），作为持仓溯源信息；不再每日滚动改写为 D4/D5。

`pool_type` 表示身份：

```text
watch      # 观察区，未买入
position   # 持仓，已买入
```
