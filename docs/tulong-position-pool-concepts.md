## 背景

0527D3 中用户实际买入：

| 代码 | 名称 | 买入价 | 数量 | 金额 |
|---|---:|---:|---:|---:|
| 600863 | 华能蒙电 | 6.170 | 600 | 3702.00 |
| 603303 | 得邦照明 | 25.440 | 200 | 5088.00 |

问题由此产生：原先把 `0528D4` 近似等同为未买入强势观察，但当前规则已收敛：D4 不再找未买入参与机会，只管理已有持仓。

## 当前确认共识

### 1. D3 统一使用“观察”，不再使用“候选”

用户已确认：候选和观察含义接近，统一为“观察”。后续用户可见概念不再出现“候选区”。

统一命名：

```text
0528D3观察区
```

工程字段统一使用：

```text
pool_type = watch
```

不再使用：

```text
candidate
候选区
```

如果历史文件名或旧文本里出现 `candidate`，只视为旧遗留/生成原因，不作为新概念继续使用。

### 2. D3 是否需要分区：需要

这里定稿：**D3 也需要分区**。

原因不是因为 D3 有多个策略池，而是因为买入动作会改变提醒目标：

- 未买入时：关注是否进入买点区、是否回收观察价、是否跌破失效；
- 已买入后：关注成本、浮盈浮亏、止损、防守线、尾盘是否处理。

因此，D3 内部也应分：

```text
D3观察区：未买入，寻找买点/回收/失效
D3持仓区：已买入，管理当天持仓风险
```

这不是“候选 vs 买入”，而是：

```text
观察 vs 持仓
```

### 3. D4 只保留持有区

D4 不再分未买入观察区，只保留已买入后的持有区：

```text
D4持有区：已买入，管理次日止损/止盈/减仓/退出
```

原来的未买入 D4 强势观察不再作为主命名，也不再作为内部观察逻辑继续使用。未买入的 D3 观察票次日不进入 D4。

## 明日监控池换日硬规则

明天 `0528` 的盘中提醒不得继续使用 `0527D3` 的观察或持仓数据。

固定规则：

```text
08:50 切池：生成/写入 0528D3 / 0528D4 的 active watchlist
09:05 守门：确认 active watchlist 里的 stage 全部是 0528D3 或 0528D4
09:25 盘中 watchdog：再次检查 stage 日期前缀，不匹配则暂停本轮提醒并输出错误
```

也就是说，即使 08:50 或 09:05 任务失败，盘中 watchdog 也不能静默继续使用：

```text
0527D3观察区
0527D3持仓区
```

## 落盘 CSV 如何区分 D3/D4 与分区

### 1. 源文件命名

源文件仍按日期+D几和时间戳区分：

```text
0528D3_<原因>_HHMMSS.csv
0528D4_<原因>_HHMMSS.csv
```

`<原因>` 只说明来源，不参与优先级，例如：

```text
scan
manual_review
position_rollover
after_close_review
```

优先级只看：

```text
0528D3 / 0528D4 + HHMMSS
```

### 2. active watchlist 字段区分

最终盘中实际读取的是：

```text
data/watchlists/tulong_active_watchlist.csv
```

关键字段：

```text
stage       # 0528D3 / 0528D4
pool_type   # watch / position
source_file # 来自哪个源CSV
```

因此三类池可以在同一个 active CSV 中区分：

| 用户概念 | stage | pool_type |
|---|---|---|
| 0528D3观察区 | 0528D3 | watch |
| 0528D3持仓区 | 0528D3 | position |
| 0528D4持有区 | 0528D4 | position |

### 3. 持仓字段

持仓区额外需要：

```text
entry_date
entry_stage
entry_price
quantity
sellable_quantity
cost_amount
position_status
```

## D3持仓 vs D4持仓：提醒指标不同

用户确认一个关键差异：

```text
D3持仓：当天买入，T+1 限制下不能卖，只能考虑加仓/风控观察
D4持有：次日已可卖，可以卖，也可以加
```

因此，`D3持仓区` 和 `D4持有区` 的提醒指标不能一样。

### D3持仓区提醒目标

D3持仓区的核心不是“卖不卖”，而是：

```text
是否守住买入逻辑
是否允许加仓
是否触发风险标记
尾盘是否需要记录隔夜风险
```

D3持仓区应提醒：

```text
成本价
现价相对成本盈亏
原始买点区/观察价
失效位/防守线
是否回到可加仓区
是否跌破失效位但因 T+1 只能标记风险
尾盘收盘位置：强、弱、接近失效、隔夜风险
```

D3持仓区动作语言：

```text
可加仓观察
不加，持有观察
风险标记，明日优先处理
尾盘弱，记录隔夜风险
```

避免使用：

```text
卖出
减仓
止盈
```

因为 D3 当天买入后不能卖。

### D4持有区提醒目标

D4持有区进入可卖日，核心是：

```text
是否卖出/减仓/止损/止盈
是否仍允许加仓
是否转弱退出
```

D4持有区应提醒：

```text
成本价
现价相对成本盈亏
可卖数量
止损/失败线
止盈/减仓区
是否突破确认
是否回踩承接可加仓
是否跌破失败线应处理
尾盘是否留仓
```

D4持有区动作语言：

```text
止损/退出
减仓/止盈
继续持有
回踩承接可加仓观察
突破确认后持有或加仓观察
尾盘不强则降低仓位
```

### 对比表

| 阶段 | pool_type | 能否卖 | 能否加 | 主要提醒 |
|---|---|---:|---:|---|
| D3持仓区 | position | 否 | 是 | 成本、防守线、加仓区、隔夜风险 |
| D4持有区 | position | 是 | 是 | 止损/止盈/减仓、加仓、是否留仓 |

## 建议概念模型

### 总结构

```text
D3
├─ D3观察区：未买入，寻找买点/回收/失效
└─ D3持仓区：已买入，管理当天持仓风险；不能卖，只能加/标记风险

D4
└─ D4持有区：已买入，管理次日止损/止盈/减仓/退出；可卖也可加
```

### 用户可见命名

```text
0528D3观察区
0528D3持仓区
0528D4持有区
```

### 工程字段建议

`stage` 表示阶段：

```text
0527D3
0528D4
```

`pool_type` 表示身份：

```text
watch      # D3观察区，未买入；D4 不使用 watch
position   # 持仓区，已买入
```
